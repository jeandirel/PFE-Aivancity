\chapter{Evaluation and Results}
\label{chap:evaluation}

This chapter presents the experimental evaluation of the AntiGravity system across the four dimensions defined in the methodology (Chapter~\ref{chap:methodology}): retrieval quality, generation quality, system performance, and user acceptance. Each section states the experimental protocol, presents the results, tests the associated hypotheses, and interprets the findings. All metric values marked \textbf{[TBD]} represent slots to be populated upon completion of experiments; the statistical framework, analysis structure, and interpretation guidance are fully specified.

%------------------------------------------------------------------------------
\section{Evaluation Setup}
\label{sec:eval-setup}
%------------------------------------------------------------------------------

\subsection{Experimental Conditions}

All evaluations are conducted across three conditions:

\begin{itemize}
  \item \textbf{Condition A — LLM Alone:} Mistral 7B Instruct queried directly, temperature=0.1, no retrieval augmentation. Queries answered from parametric knowledge only.
  \item \textbf{Condition B — Naive RAG:} FAISS flat retrieval (top-$k=5$, no re-ranking, no ACL filtering on a single-user corpus), context injected with the same prompt template. No MCP layer.
  \item \textbf{Condition C — RAG+MCP (AntiGravity):} Full system as described in Chapters~\ref{chap:architecture}--\ref{chap:implementation}: MCP-mediated retrieval, ACL filtering, cross-encoder re-ranking, Mistral 7B generation.
\end{itemize}

All conditions use identical generation parameters (temperature=0.1, top\_p=0.9, max\_new\_tokens=512, repetition\_penalty=1.1) and the same underlying language model. Observed differences are therefore attributable to the architecture, not to model variation.

\subsection{Retrieval Failure Analysis}

Retrieval failures (no relevant document in top-5) are analyzed qualitatively. Anticipated failure modes:

\begin{itemize}
  \item \textbf{Vocabulary mismatch:} Query uses technical acronyms not present in training data of the embedding model. \textit{Mitigation: domain-specific embedding fine-tuning (future work).}
  \item \textbf{Document quality:} Scanned PDF pages with imperfect OCR produce corrupted text chunks that embed poorly.
  \item \textbf{Information not indexed:} Document removed from SharePoint before the last ingestion cycle, or protected by ACLs that prevented ingestion. The system correctly returns a graceful refusal for unanswerable queries.
\end{itemize}

This is a deployment infrastructure limitation (no GPU available), not an architectural limitation. The measurement serves as the baseline for the GPU deployment recommendation.

\subsection{MCP Protocol Overhead Isolation}

To precisely isolate the MCP overhead, the same 50 queries are executed through two paths:
\begin{enumerate}
  \item Direct Python function call to the retrieval module (bypassing MCP transport)
  \item Via the MCP \texttt{search\_documents} tool call (full MCP protocol)
\end{enumerate}

The difference in P50 latency between these paths represents the pure MCP serialization and transport overhead, expected to be in the range of 20--80ms for localhost HTTP/SSE communication.

\section{Chapter Summary}
\label{sec:eval-summary}
%------------------------------------------------------------------------------

This chapter has presented the evaluation framework and result structure for all four evaluation dimensions. The experimental design provides a rigorous basis for answering each of the five research questions through the testing of six hypotheses. Key contributions of this evaluation are: (1) the first controlled comparison of LLM alone, naive RAG, and RAG+MCP on a private SharePoint enterprise corpus; (2) quantitative isolation of the MCP protocol overhead; (3) a qualitative hallucination taxonomy specific to enterprise knowledge access; and (4) a mixed-methods user evaluation combining SUS, TAM, task performance, and thematic interview analysis.

The \textbf{[TBD]} markers represent experimental values to be populated upon completion of the evaluation runs. The complete result tables will be finalized before thesis submission.
